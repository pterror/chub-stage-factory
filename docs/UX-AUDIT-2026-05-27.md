# UX Audit — chub-stage-factory — 2026-05-27

A three-lens UX audit of the chub-stage-factory repository, scoped to:

1. **Stage-author UX** — someone (human or LLM) who clones this repo to ship one Chub stage. Walks through co-design, sees scripts, sees errors.
2. **Composer-author UX** — someone extending `src/lib/patterns/` (or `src/lib/patterns/synergy/`). Has to learn the convention from existing files and produce one that fits.
3. **End-user UX** — the player loading a deployed stage and taking their first turn. Studied through the 8 example stages.

Frameworks applied (see `~/git/rhizone/github-io/docs/{interaction-graph,affordance-opacity,affordance-surfaces,affordance-types}.md`):

- **Interaction graph** — every affordance is an edge; failures are missing/hidden/orphaned edges, bottlenecks, sparse regions.
- **Affordance opacity** — what the system hides about what you can do. Tribal knowledge cost.
- **Affordance surfaces** — Miller's Law (≤7 simultaneous items) at every level. Removal over prioritization.
- **Affordance types** — commands vs gestural vs ambient vs navigational vs directional vs data-entry. Different surfaces render different types.

The audit was specifically commissioned to answer: **what blocks Phase 5 manual browser verification?** That answer is in the Phase 5 Blocker List section. The rest of the audit is the substrate that makes that list legible.

## Personas

These three are derived from observable repository structure: the `CLAUDE.md` two-phase workflow (Phase 1 co-design / Phase 2 autonomous build), the `src/lib/patterns/` + `src/lib/patterns/synergy/` shape, and the 8 deployed `examples/`. Names are arbitrary; goals and contexts are not.

### P1. "Yuna, the weekend stage author"

Yuna found Chub a month ago, plays roleplay stages a few times a week, and watched a tutorial video on building one. She has just enough programming background to know what TypeScript is and how to run a yarn command. Saturday afternoon she clones chub-stage-factory because she wants to build "an inn-keeper stage where the player can buy drinks and ask gossip — like the bar-keeper stage but in her own setting."

**Context:**
- Reads `README.md` first, then `CLAUDE.md` if she gets stuck.
- Will spend ~2 hours before deciding whether to continue or bounce.
- Wants to *see something on screen* in the first 20 minutes.
- Does not know what a "primitive" or a "composer" is; doesn't speak the library vocabulary.
- Will Cmd-click into example stage code to learn by example.
- Doesn't have a CHUB_AUTH_TOKEN, doesn't want to think about deployment yet.

**Goal:** A running local preview of a recognisable inn scene with one NPC and one verb that does something visible. From there, she'll iterate.

**What she'll touch:** `README.md`, `CLAUDE.md`, `DESIGN.md` (via Claude Code), `src/Stage.tsx`, `yarn dev` URL, eventually `examples/inventory/Stage.tsx` or `examples/world-primary/Stage.tsx` for reference.

### P2. "Rin, the composer hunter"

Rin is an experienced TypeScript developer who has built three Chub stages before, all by copy-pasting from `extension-template`. They found chub-stage-factory through a Discord link and were drawn in by the COMPOSITION.md framing — "patterns instead of frameworks, primitives are orthogonal." They want to add a new composer: `patterns/conversation-tree.ts`, a dialogue-tree composer that wires `dialogue.ts` + `triggers.ts` into a Twine-shaped state machine. It's not in PATTERNS.md yet — they think it should be, and want to test the claim that "if a thing reduces, it's permanently gone."

**Context:**
- Reads `src/lib/COMPOSITION.md`, `src/lib/README.md`, `src/lib/CONVENTIONS.md` before touching any code.
- Picks 3 existing composers to study the convention. Will their reading converge on a learnable pattern?
- Wants to PR upstream eventually but will keep a local branch first.
- Cares about `bun run lint`, `bun run test`, `bun run build` passing — has seen messy repos before and is suspicious.
- Knows what an `@experimental` marker is and will check whether 0-caller primitives suggest a graveyard.

**Goal:** Ship a composer that (a) builds, (b) passes lint+test, (c) follows whatever the convention turns out to be, and (d) is sufficiently distinct that no existing composer subsumes it.

**What they'll touch:** `src/lib/patterns/synergy/cache-by-key.ts`, `src/lib/patterns/inventory.ts`, `src/lib/patterns/scene.ts` (for reference), then a new `src/lib/patterns/conversation-tree.ts` + an `examples/conversation-tree/` directory + an entry in `examples/registry.ts`.

### P3. "Tomás, the curious player"

Tomás opened the Chub homepage, browsed for stages, and the recommendation engine surfaced `world-primary` and `composite-showcase` (because they share tags with stages he played last week). He clicks `world-primary` first. He has never seen a stage that has its own UI (most stages he's played are pure prose-injection). He's going to give it 60 seconds to make sense or he'll click away.

**Context:**
- Brings zero technical background. Reads no docs.
- Sees the stage embedded in Chub's chat window. First impression is purely visual.
- Has muscle memory from text-only chat: type, press Enter, read reply.
- A button is a button. He'll click anything that looks clickable.
- Won't notice subtle UI affordances (resize handles, hover-only tooltips) unless they're labelled.
- If anything visibly fails — empty panel, frozen button, garbage prose — he bounces.

**Goal:** Have a couple of minutes of light fun. Maybe come back tomorrow if it sticks.

**What he'll touch:** the rendered DOM of `examples/world-primary/Stage.tsx` (or whichever stage Chub serves). Specifically the `ActionSurface` verbs, the `FreeformInput`, the `ScenePane`, the Chub-native message box.

### Findings — Lens 1

**GREEN — Phase 1 framing is correct.**

`CLAUDE.md` lines 67-85 frame Phase 1 as a co-design conversation that fills DESIGN.md before any code is written. The "bar for concrete enough" is named explicitly. Yuna does not need to know what a primitive is; Claude can name the relevant primitive for her based on her described mechanic. This is the highest-leverage UX choice in the repo and it works.

**GREEN — Reference examples are properly indexed.**

`examples/README.md:45-56` lists all 8 examples with their mechanic and the PATTERNS.md recipe they exemplify. `examples/registry.ts:46-136` provides one-line descriptions of what each demonstrates. Yuna can scan this table in 30 seconds and pick the closest match. This is a *good affordance surface* per the surfaces framework — 8 items is right at Miller's limit, the labels are human-readable, and the surface is one rendering of the underlying registry.

**YELLOW — README is too thin to function as front door.**

`README.md` is 46 lines (`README.md:1-46`). It mentions "clone the repo, talk through Phase 1, run /loop" — but for Yuna, this assumes she already knows what a slash command in Claude Code is. There is no minimal "what you'll see" — no screenshot, no example output, no "if this works, you'll have X on screen." The README is a navigational affordance pointing into CLAUDE.md, but the destination is 266 lines and primarily addresses agents, not Yuna.

**YELLOW — `DESIGN.md` skeleton has no examples in it.**

`DESIGN.md:1-83` is a blank-table skeleton. Yuna staring at "Tags" with no example values doesn't know what shape they should be (Chub tag slugs? freeform? both?). "Position: ADJACENT | NONE | COVER | FULLSCREEN" is a comment-hint; she has no idea what each visually means. The framework lens: this is a high-opacity data-entry surface. Fix would be a parallel `DESIGN.example.md` showing the *world-primary* DESIGN as a worked example.

**YELLOW — `yarn dev` vs `bun run dev` vs `yarn install` inconsistency.**

`README.md:15-16` and `CLAUDE.md:153-158` both tell Yuna to run `yarn install` then `yarn dev`. `package.json:17-18` declares `bun scripts/run-stage.mjs` for smoke tests and `bun install` is referenced in DECISIONS.md §9. `flake.nix` (not read but referenced) provides `node@21.7.1 + yarn`. Yuna who follows the README will install yarn; she will not have bun, and the smoke runner will fail. The repo uses both. CI uses bun (DECISIONS.md §9). Local dev docs say yarn. Pick one.

**RED — Build script error mode is destructive on interrupt.**

`scripts/build-example.mjs:36-66` performs:
1. `cpSync(publicDir, backup)` — backup
2. `rmSync(publicDir, ...)` — wipe public/
3. copy example's assets in
4. `vite build`
5. `finally { rmSync(public) + cpSync(backup, public) }` — restore

If Yuna Ctrl-Cs during step 3 or 4, the `finally` runs and restores. Good. But if the **node process is killed (-9, OOM, crash)**, public/ is left wiped — the next invocation will back up an empty directory, then permanently destroy the real public/ on the third invocation when temp dirs roll over. There is no checksum, no lock, no "is public/ a known template state?" check. For Yuna, this manifests as: ran build twice on a flaky laptop, public/ is now empty, fresh-clone needed. There's no STATUS.md trace of this hazard. Cite: `scripts/build-example.mjs:32-67`.

**RED — Phase 2 has no human-facing progress signal.**

`CLAUDE.md:73-86` describes Phase 2 (`/loop`) as autonomous: it picks the next unchecked STATUS.md item, edits, builds, commits, pushes, repeats. Yuna is supposed to leave this running. But the *user-visible signal of progress* is: STATUS.md updates and git log. If she opens `yarn dev` mid-loop, she'll see whatever last-committed render exists — possibly mid-feature, possibly broken-but-typechecks. There is no "Phase 2 is at task 4 of 9, current build status green" surface. Yuna will tab-out and tab back wondering if anything is happening. This is the *invisible state* failure mode from the interaction graph framework.

**RED — `/loop` exit conditions are buried.**

`CLAUDE.md:85` says "Exit the loop when STATUS.md shows all tasks complete and the latest push passed the deploy workflow." Yuna does not know how to check the deploy workflow status. There is no `bun run status` or `node scripts/check-deploy.mjs` that tells her "you're done, or you're stuck on X." She has to know `gh run list` exists. This is tribal knowledge for power users.

**YELLOW — Promote-example exists but isn't discoverable from the front door.**

`scripts/PROMOTE.md` describes the `promote-example.mjs` tool, which is genuinely Yuna's exit ramp: "I built my stage in this monorepo, now copy it out so I have a clean per-stage repo." But `README.md` doesn't mention it, `CLAUDE.md` doesn't mention it, `examples/README.md` doesn't mention it. The only way Yuna finds it is `ls scripts/` or grepping. Orphaned affordance.

**YELLOW — `DESIGN.md` blank skeleton lives at the root.**

`DESIGN.md:1-83` is in the repo's root, alongside `STATUS.md`, `TODO.md`, `DECISIONS.md`, `CLAUDE.md`, `README.md`. Six top-level meta-files. Per Miller, this is at the upper bound; per the surfaces framework, it's a scanning-vs-searching boundary. A first-time visitor sees `ls` output and has to *read each filename* to triage. A subfolder `meta/` or naming convention (`PHASE1-DESIGN.md`, `PHASE2-STATUS.md`) would chunk visibly.

### Findings — Lens 2

**GREEN — Composer header comments are a strong convention.**

Every composer file under `src/lib/patterns/` opens with a doc-comment block following the schema `WHAT / WHY / SHAPE`. See `src/lib/patterns/inventory.ts:1-28`, `scene.ts:1-43`, `freeform-pipeline.ts:1-40`, `grafting.ts:1-21`. This is the convention Rin will pattern-match to. After reading 3, the shape is internalised. This is the **clean graph** outcome the interaction-graph framework praises: no filtering needed, the convention is the convention. The 33-composer batch (DECISIONS §13) maintained this; Rin can read any one and produce a 4th that fits.

**GREEN — Synergy patterns have an enforced shared return shape.**

`src/lib/patterns/synergy/types.ts` declares `ComposedSubsystem<S>` with `state`, optional `contributors`, `hooks`, `shards`. All 22 synergy composers import this. `cache-by-key.ts:38-60` and `seed-from-player.ts:37-66` both return the same shape with a domain-specific extra method. Rin writing a 23rd synergy composer has a typed contract; if they don't return `ComposedSubsystem<…>`, TSC fails. This is *exactly* the affordance-types insight: when the type system enforces the surface, discovery becomes free.

**GREEN — `bun run build` / `lint` / `test` / `test:smoke` exist and are wired into CI.**

`package.json:13-26` exposes all four. `.github/workflows/build-examples.yml` (not read this audit but per DECISIONS §9) runs all four. Rin's "did I break anything?" loop is fast and credible.

**YELLOW — Top-level `src/lib/patterns/` (28 composers) violates Miller's Law as a surface.**

A flat directory listing of 28 composers — `bulk-tick`, `dialogue`, `effects`, `faction`, `form`, `form-collection`, `grafting`, `inventory`, `lineage`, … — exceeds the working-memory cap by 4×. Rin will search rather than scan. PATTERNS.md groups them by recipe number, which helps; but the file tree does not. Compare to the synergy subdir (22 files, also over) — at least synergy has the type-shared `types.ts` to anchor on, plus the source-attribution comment ("Source: SillyTavern WI / NovelAI Lorebook keys / AID Story Cards" — see `cache-by-key.ts:9-12`). The top-level patterns dir has no such grouping. A subfolder layout (`patterns/world/`, `patterns/combat/`, `patterns/character/`, `patterns/synergy/`) would chunk to ~7 per group.

**YELLOW — Composer naming inconsistent between "extracted" and "newly built."**

DECISIONS §13 distinguishes extracted (lifted from examples) vs newly-built. Extracted ones (`inventory.ts`, `effects.ts`, `turn-combat.ts`, `body-transformation.ts`) return `*Bundle` types and use `pattern.buildBeforePrompt(msg, bound)`. Newly built ones like `grafting.ts` return a different-named object with `subsume`/`inject`/`replace` methods directly. Synergy ones return `ComposedSubsystem<S>`. This is **three return shapes for three vintages**. Rin asking "what shape should my composer return?" gets three valid answers. PATTERNS.md does not name the discriminant. See: `src/lib/patterns/inventory.ts:61-75` vs `src/lib/patterns/grafting.ts` vs `src/lib/patterns/synergy/types.ts:48-59`.

**YELLOW — `withPersistence` HOC is the cleanest path but isn't documented at the top of patterns/.**

`examples/inventory/Stage.tsx:35` uses `extends withPersistence<…>()` — a class-factory HOC. DECISIONS §1 says all 8 examples use it. But the docstring on `withPersistence` itself is in `src/lib/persistence/with-persistence.ts` (not read), and `src/lib/PATTERNS.md` §0 still shows the *raw* `PersistenceStore` + `bindStore` ceremony (`PATTERNS.md:22-65`), not the HOC. Rin reading PATTERNS.md will write 40 lines of persistence boilerplate that all 8 examples have already collapsed into one HOC call. The recipe doc is out of date relative to the examples.

**YELLOW — `@experimental` markers are honest but mark dead-end edges.**

DECISIONS §7 marks `chat-window.ts`, `embeddings.ts`, `3d/scene.tsx`, `3d/loader.tsx`, `3d/use-three-handle.ts` as `@experimental — used by 0-1 callers`. Rin building a new synergy composer that depends on `embeddings.ts` is building on a marked-experimental primitive whose own corrected-caller-count (DECISIONS §7 re-audit) is *zero*. There's no "do not depend on experimental" rule. The affordance-types lens: these are gestural affordances (low discoverability) mixed in with command affordances (high discoverability) on the same surface without distinction. Either commit (un-experimental) or delete; sitting in `@experimental` indefinitely is the worst of both worlds.

**RED — PATTERNS.md does not list every shipped composer.**

`src/lib/patterns/` contains 28 composers; `src/lib/patterns/synergy/` contains 22. `PATTERNS.md` (read only the first 200 lines this audit) starts with recipes 0, 0a, 0b, 0c, 1, 2 — recipe 1 is "Inventory" and points at raw `Inventory` primitive code, not `inventoryPattern`. Recipes 3-7 likely follow; whether each shipped composer has a documented recipe is unverified but doubtful given the 33-composer batch from a single afternoon. Rin trying to "find the recipe before writing the composer" hits a sparse documentation graph. Missing edges from PATTERNS.md to the composer files. Cite: `src/lib/PATTERNS.md:106-141` (recipe 1 shows raw primitive, doesn't mention `inventoryPattern`).

**RED — Composer-author has no scaffolding tool.**

To add `conversation-tree.ts` Rin must by hand:
1. Create the file with the `WHAT/WHY/SHAPE` header.
2. Decide which return shape (`*Bundle` vs `ComposedSubsystem<S>` vs ad-hoc).
3. Wire imports from `../intent`, `../trigger`, `../timeline`, etc.
4. Create `examples/conversation-tree/Stage.tsx` + `test-init.json` + `chub_meta.yaml` + `scenario.yaml` + `characters/` (per `examples/README.md:24-28`).
5. Add entry to `examples/registry.ts:46-136`.
6. Update `PATTERNS.md` (if maintained), `REFERENCE.md`, `ROADMAP.md`.

No `bun run new:pattern <name>` or `node scripts/scaffold-composer.mjs`. This is reasonable for a single-author repo; for a "library positioning itself as accepting community composers," it's an undocumented barrier. See `scaffolding/` directory in the ecosystem-docs `CLAUDE.md` for the pattern that *should* exist here.

## Lens 3 — End-user UX (the 8 examples)

Tomás is the persona. Each example is evaluated on: **first-turn experience**, **what he sees in the rendered DOM**, **affordance types available**, and **whether the UI explains itself within 60 seconds**.

### 3.1 inventory — Pak the pack-rat shopkeeper

`examples/inventory/Stage.tsx:89-111` renders:
- Title: `Pak's stall — tick {N}`
- Table: spot | contents (with disorder + access scores in dim text).

**First turn for Tomás:** He types into Chub's chat box: "What's on the counter?" Pak's LLM responds via the `STAGE_DIRECTIONS.prefix` (`inventory/Stage.tsx:29-33`) — Pak is "POV-adjacent shopkeeper", the LLM gets the spots+items as observation data, and writes prose like "Pak shrugs at the counter — a brass compass and a ledger…"

**Affordance types in DOM:** ambient only. No buttons, no inputs (other than Chub's own message box). The rendered table is a *status display* — Tomás reads it, the table updates after each turn, but he cannot click anything to act on it.

**Verdict (Tomás):** GREEN as a *display*. The table is legible, the disorder/access scores are flavour data he can ignore. The interaction graph is entirely "type in Chub box → bot replies → table updates" — one path, low surprise.

**Hidden problem:** the `access` and `disorder` scores are uninterpreted numbers. `a=0.83` next to "brass compass" — what does 0.83 mean? Tomás guesses higher = easier? Lower? No legend. Ambient affordance failing at attention (per affordance-types framework).

### 3.2 effects — Klio the apothecary

`examples/effects/Stage.tsx:86-113` renders: tincture list, active effects with remaining duration and magnitudes JSON, event log `<pre>`.

**First turn:** Tomás types "What can I drink?" The LLM (per prefix `effects/Stage.tsx:67-73`) emits `<apply>adrenaline</apply>` in its reply, the tag is stripped, an effect ticks onto the panel.

**Affordance types in DOM:** ambient only. JSON dump of magnitudes (`{"dodge":0.2,"damage":1}`) is a *developer surface* leaking to the user.

**Verdict:** YELLOW. The JSON magnitude dump (`stage.tsx:99-105`) is high-opacity for a player. He sees `{"dodge":0.2,"damage":1}` and asks "are these visible to my character? are these stats I should track?" The frame is *developer-debug*, not *player-facing*.

### 3.3 turn-combat — duel on the steps

`examples/turn-combat/Stage.tsx:127-153` renders: title with round + queued choice, combatants table (HP/AP/armor/effects), last-round events `<pre>` (raw JSON).

**First turn:** Tomás types "I attack." The LLM emits `<action>swing</action>` (per prefix `turn-combat/Stage.tsx:108-113`), the combat round resolves on `afterResponse`, the table updates, the events pre block shows JSON like `{"kind":"hit","attacker":"pc","target":"duellist","damage":4}`.

**Affordance type leak:** events `<pre>` raw JSON. Player sees ground-truth combat log but in machine format.

**Verdict:** YELLOW. Combat *works*; the LLM does the prose. But the JSON log is dev-surface noise. Also: queued-choice is *invisible state* — Tomás has typed "I attack" but the UI shows `you queued: swing` — he doesn't know what queued means, doesn't know he can override.

### 3.4 tits-body — Vey the alchemist

`examples/tits-body/Stage.tsx:103-125` renders: body slot table, active transformations list with elapsed/duration, snapshots line.

**First turn:** Tomás types "give me a cat tail." LLM emits `<drink>cat_tail</drink>`. The body table updates over multiple turns as the trajectory ramps (nub → stubby-tail → full furred prehensile tail).

**Affordance type:** ambient. No buttons. To restore baseline: `<restore>baseline</restore>` — this is named only in `STAGE_DIRECTIONS.prefix` (`tits-body/Stage.tsx:83-86`). Tomás would have to know to ask "can you restore baseline?" — *the affordance is hidden in the LLM's system prompt, not in the UI*.

**Verdict:** YELLOW-leaning-RED. The UI shows internal tag strings (`furred, prehensile-mild, tail-cat`) directly. These are slot tags, not prose. Tomás reading "tail-cat" wonders "is that a typo for 'cat tail'?" The dev-vs-player surface confusion is acute here because the body changes are exactly what the player wants to see, but in player-language not author-language.

### 3.5 cyber-slots — Dr. Cull the ripperdoc

`examples/cyber-slots/Stage.tsx:133-156` renders: body tags, equipped items with fit (comfortable/tight/broken color-coded), violations as raw JSON.

**Violations pretty-print:** `<pre>{JSON.stringify(this.cyber.loadout.checkAllConstraints(), null, 2)}</pre>` (`cyber-slots/Stage.tsx:152`). This is the *most* developer-surface-leaking render in any of the 8 examples. Tomás sees:

```json
[{"source":"deckjack","constraint":"!flesh-only","failedTerms":["flesh-only"],"context":{}}]
```

**Verdict:** RED. The clinical-prose framing ("Dr. Cull is the ripperdoc; the player is the patient") clashes badly with a violations panel that reads like a TSC error. Either the violations should be plain-English ("your deckjack rejects fleshweave"), or hidden, or color-coded chips. As shipped, it's debug output exposed to the patient.

### 3.6 physics — Mara's atelier

`examples/physics/Stage.tsx:96-113` renders: SVG of room (200×120 viewport scaled to 400×240), grey rectangles for obstacles, cyan/orange dots for trajectory steps (orange = bounced).

**First turn:** Tomás types "throw something at the workbench." LLM emits `<throw>10,10,5,-3</throw>` (per prefix `physics/Stage.tsx:75-78`). The trajectory dots appear, the "hit" list updates.

**Affordance type:** ambient + a *visual* gestural one (the SVG) — but the user cannot interact with the SVG. He can't click on the room to throw. The SVG is a rendering of physics state, not an input surface.

**Verdict:** GREEN as visual feedback. YELLOW as interactive: a click-to-throw on the SVG would be 30 lines of code and would transform this from "watch the bot do physics" to "play physics." The current design is a *display affordance only*; the agent is the only thing that can act on the world.

### 3.7 realtime-combat — arena drone fight

`examples/realtime-combat/Stage.tsx:139-156` renders: SVG arena (240×160 → 480×320), player as cyan circle, drones as red circles (grey when dead), bullets as yellow circles, events `<pre>` below.

**First turn:** Tomás types "shoot the closest drone." LLM emits `<shoot>1,0</shoot>` (per prefix `realtime-combat/Stage.tsx:113-116`); 5 physics ticks advance; events log shows hits/misses; drones spawn every 3 turns.

**Affordance type:** identical to physics — SVG is display-only.

**Verdict:** GREEN as visual; YELLOW as interactive. Same observation: click-on-arena-to-shoot would unlock a *direct manipulation* affordance. Currently Tomás has to describe directions in prose and the LLM decides numbers — a high-friction loop for an arcade-style mechanic.

### 3.8 world-primary — the "just-good RP frontend"

`examples/world-primary/Stage.tsx:596-694` renders a **full-page 3-column layout**: world state panel (left), action surface + scene pane + freeform input (centre), chat log sidebar (right).

This is the only example with **command affordances** in the DOM: `ActionSurface` renders verb buttons. `world-primary/Stage.tsx:539-594` derives verbs from world state: "Go north → The Ember Inn", "Talk to Elder Mira", "Examine Map Fragment", "Take Map Fragment".

**First turn:** Tomás sees buttons. He clicks "Go north → The Ember Inn." The button's `onClick` is `() => { this.isProcessing = true; }` (`world-primary/Stage.tsx:551-554`) — *it sets a flag and does nothing else*. The verb buttons do not actually drive state changes. The freeform input also has an empty `onSubmit` (`world-primary/Stage.tsx:676-683`). The comment admits this: "In a Chub stage the player's message routes through Chub's chat box → beforePrompt. The FreeformInput here is a UI affordance whose submit the stage author would wire to Chub's message injection API (not available in this adapter)."

**Verdict:** RED. The most ambitious example is also the most broken from a player's perspective. Buttons exist, look clickable, don't work. The world-primary design intent (the whole point of `FRONTEND-SHAPE.md`) is "structured verbs as fast path" — the verbs are rendered but the path is severed. Tomás clicks "Go north", nothing happens, the verb button doesn't even visually depress. Worse than no button: a *visible-but-dead* affordance, which the affordance-opacity framework calls "orphaned affordances."

**Additional surface problems in world-primary:**
- 3-column layout assumes wide viewport. On a phone or narrow Chub iframe, it will overflow or hide columns. No responsive collapse.
- The action surface renders `verbs.length` buttons — for a room with exits N/E/S/W, 3 NPCs to talk to, 4 items to examine and take, that's potentially 4+3+8 = 15 buttons. Over Miller's limit. `ActionSurface` uses `columns={3}` so it tiles; that helps scan-vs-search but doesn't change the cognitive load.
- The chat log sidebar duplicates Chub's own chat history. Players see "their last message" twice on screen, in different positions.

### 3.9 composite-showcase — Maven's clinic

506-line stage combining body+equipment+inventory+combat. Render not read in detail; presumed to be the union of cyber-slots + tits-body + inventory + turn-combat panels, each rendering the same dev-surface issues identified above. The "PlaceholderRegistry demo" (LLM-invented cyberware via `<invent>` tag, `composite-showcase/Stage.tsx:16-23`) is the most novel affordance in the suite — but it's again hidden in the system prompt, not surfaced to the player.

**Verdict (presumed):** YELLOW — inherits every problem from the four examples it composes, plus the invent-cyberware feature is invisible.

## Cross-cutting findings (framework lenses)

### Interaction graph topology

**Bottleneck node: the LLM tag emission.** Every example except world-primary routes player action through prose-the-LLM-emits-tags-in. `<drink>cat_tail</drink>`, `<apply>adrenaline</apply>`, `<install>install_neural_port</install>`, `<shoot>1,0</shoot>`. This is a *single point of navigation*: if the LLM fails to emit the tag (or invents a name), the entire action path is dead. There's no fallback verb-button affordance (world-primary tried, see Lens 3.8). This is the **bottleneck node** failure mode named in the interaction-graph framework. The library has `intent.ts` and `freeform-pipeline.ts` precisely for routing around this; they're used in exactly one example.

**Orphaned subgraphs: `@experimental` primitives.** `embeddings.ts`, `chat-window.ts`, the entire `3d/` subsystem (0 external callers each, DECISIONS §7). These are nodes in the graph with no incoming edges from real stages. Per the interaction-graph framework, they should either get an edge (integrated into an example) or be removed (no orphan nodes).

**Sparse region: composer-author flow.** From "I have a pattern idea" to "I have a composer + example + lint+test green" the graph traversal is 6+ steps with no automation. See Lens 2 finding RED-2 (no scaffolding tool).

**Dense region: persistence boilerplate.** The original `PersistenceStore + bindStore + shard + chubTreeHistory + forbidBranching + snapshotHistory + mergeResponses` cluster was a dense subgraph — 7-8 imports per stage to wire one concern. DECISIONS §1 (`withPersistence`) collapsed it to one import + `extends withPersistence<…>()`. Good architectural reduction.

### Affordance opacity scorecard

| Surface | Opacity | Tribal knowledge to use |
|---------|---------|--------------------------|
| `/design-stage` slash command | High | Must know `.claude/commands/` exists |
| `/build-stage` + `/loop` flow | High | Must know slash commands are not documented in README |
| `bun run test:smoke` vs `yarn` | Medium | Requires reading DECISIONS to know bun is the actual runtime |
| `node scripts/promote-example.mjs` | High | Only discoverable by `ls scripts/` |
| `withPersistence` HOC | Medium | Documented in DECISIONS §1 but PATTERNS.md shows the old way |
| LLM action tags (`<drink>`, etc) | High to player | Only in `STAGE_DIRECTIONS.prefix` — player UI never names them |
| Stage render dev-leak (JSON dumps) | High | Player doesn't know if it's information for them or for someone else |
| Example registry entries | Low | `examples/README.md` table is good |

### Affordance surface violations (Miller's Law)

| Surface | Items | Verdict |
|---------|-------|---------|
| Root-level meta files | 6 (CLAUDE/DESIGN/STATUS/TODO/DECISIONS/README) | At limit |
| `src/lib/` (top level) | 50+ .ts/.md files | 7× over |
| `src/lib/patterns/` | 28 composers | 4× over |
| `src/lib/patterns/synergy/` | 22 composers + types.ts | 3× over |
| Example registry (`examples/`) | 8 directories | At limit (good) |
| world-primary ActionSurface | up to 15 verbs simultaneously | 2× over |
| world-primary 3-column layout | 3 columns (top level) | Good — but each column has its own surface count |
| CLAUDE.md sections | 14 H2 sections | 2× over (forces TOC search) |

### Affordance type imbalance

The 8 examples render almost exclusively **ambient** affordances (status displays). One renders **command** affordances (world-primary), and those are broken. None render:

- **Gestural** — no drag, no resize, no hover-target hits. The physics SVG could be clickable; isn't.
- **Directional** — no scroll/pan/zoom. The arena could be panable; isn't.
- **Data-entry** — no inline forms, sliders, color pickers. (The freeform input in world-primary is data-entry but its submit is unwired.)
- **Navigational** — no tabs, breadcrumbs, links. (The 3-column layout is structural, not navigational — you can't move between rooms via the UI.)

This is the affordance-types framework's central lesson: **a system that models commands will produce a system biased toward commands.** Here the bias is even narrower — it models *observations the LLM consumes*, not affordances the player operates. The render() method is conceptualised as "show the LLM's input back to the user," not "give the user a way to act."

## Mock-stage navigation tooling proposal

The user's question: should the factory ship "tooling that lets agents navigate a mocked, headless representation of the real stage codebase"?

### What already exists

- `src/runner/IframeHost.tsx` + `src/runner/protocol.ts` + `src/runner/mocks.ts` — a browser-based postMessage host with `NullMocks` / `PassthroughMocks` / `RecordReplayMocks`.
- `scripts/run-stage.mjs` — CLI smoke runner using jsdom + react-dom/server. Drives a stage through a scenario JSON, evaluates assertions on rendered HTML and on `messageState`/`chatState`.
- `scenarios/*.smoke.json` — scenario fixtures with `init`, `steps` (`before`/`after`/`set`/`call`), and `assertions` (`messageState`/`chatState`/`domContains`/`domMatches`).
- `src/runner/Inspector.tsx` — live JSON view of message/chat/init state.

These together cover: (a) deterministic headless replay, (b) browser-iframe interactive driving, (c) live state inspection, (d) mock LLM responses.

### What would mock-stage navigation tooling add?

The implied need: an agent (Claude Code in autonomous mode, or a Phase-5-style verifier) wants to "explore" a stage without running it visually. Concretely:

1. Enumerate available actions at the current state.
2. Take an action, observe state delta.
3. Repeat — like an interactive REPL over the stage's interaction graph.

`scripts/run-stage.mjs` only supports scripted scenarios. It cannot reply to "what verbs are available right now?" because **stages don't expose that as a queryable surface**. The agent would have to render to DOM, parse the HTML for buttons, and synthesise verbs back to actions — fragile.

### Proposed API shape (if shipped)

A `StageIntrospect` interface that examples implement optionally:

```ts
export interface StageIntrospect<MS> {
  /** Verbs available now, with hints for what they require/produce. */
  availableVerbs(state: MS): { id: string; label: string; requires?: string[]; produces?: string[] }[];
  /** A short text description of the current state for an agent. */
  describe(state: MS): string;
  /** Take a verb by id. Returns the resulting message text to feed back into beforePrompt. */
  invokeVerb(id: string, state: MS): string;
}
```

A CLI driver `scripts/explore-stage.mjs <example>` that:
- Loads the stage via the headless registry path.
- Calls `availableVerbs` to print the menu.
- Reads stdin for a verb id (or accepts an LLM-loop driver).
- Invokes the verb, prints `describe(state)`, repeats.

For agents specifically, the CLI would also expose a JSON-RPC mode that an external Claude Code session can drive turn-by-turn — *the agent never has to render HTML to find affordances*.

### Verdict

**The need is real, but it's the same need as fixing world-primary's broken ActionSurface.** The orphan-edge problem in lens 3.8 is the same problem: stages do not have a queryable affordance graph. They have a render() that produces an opaque DOM.

If shipped: the introspect interface is the right primitive. It would solve both Phase 5 verification (Claude can drive a stage headless turn-by-turn, asserting on `describe()` output) *and* the world-primary verb-wiring problem (the verbs the UI renders are the verbs `availableVerbs` returns; clicking calls `invokeVerb`).

**Recommendation:** ship `StageIntrospect` as an optional interface. Make world-primary the first implementer. Reuse it from `explore-stage.mjs`. This is one architectural move, not two — and it addresses both the audit's biggest finding (orphaned UI affordances) and the user's question (mock navigation tooling).

If not shipped: the existing `run-stage.mjs` + scenarios + mocks suffice for *scripted* verification. They do not suffice for *exploratory* agent-driven verification — and exploratory is what Phase 5 will need first.

## Phase 5 manual-verification blocker list

The audit's commissioning question. Concrete blockers to "open a browser, load a deployed stage, click around to verify it works":

1. **`world-primary` ActionSurface verbs are wired to no-ops.** `examples/world-primary/Stage.tsx:551-590` — every verb's `onClick` is empty or sets a flag without dispatching a message. Manual verification cannot test the verb path because there is no verb path. **Unblock:** wire `onClick` to inject a synthetic message via Chub's message API (research how that's exposed) or to call a local `processIntent` method.

2. **`world-primary` `FreeformInput` `onSubmit` is empty.** `examples/world-primary/Stage.tsx:676-683` — comment admits it. The freeform path the stage was designed around is unreachable from the UI; verification can only go via Chub's chat box. **Unblock:** same fix as #1, plus reconcile "stage UI input" vs "Chub message box" duplication (pick one).

3. **No deploy-status check command.** `CLAUDE.md:85` instructs the loop to exit "when latest push passed the deploy workflow." There is no `bun run check-deploy` or `node scripts/deploy-status.mjs`. Manual verification of "did this stage actually ship to Chub?" requires `gh run list` + `gh run view` + reading workflow YAML to find the deploy step. **Unblock:** ship a `scripts/check-deploy.mjs` that reads the latest workflow run for the current commit and prints status.

4. **No "first-load contract" assertion.** `scripts/run-stage.mjs` runs scenarios; no scenario asserts "the stage renders something non-empty on `load()` with zero state." Manual verification of a freshly deployed stage means clicking through Chub UI hoping it renders. **Unblock:** add `scenarios/<name>.first-load.smoke.json` per example — load + render + assert `domContains` on a label string.

5. **`@experimental` primitives have no integration paths.** A verifier asked "is `embeddings.ts` working in production?" cannot answer because no example uses it. **Unblock:** either delete `@experimental` primitives or write a `examples/_test-embeddings/Stage.tsx` integration that exercises each — verifier can then run the smoke against it.

These five blockers must be cleared before "manual browser verification" is a coherent activity. Without them, the verifier is hand-rolling fixes rather than verifying behavior.

## Recommendations (ordered by leverage)

Each: **what** / **why** / **who** (which persona benefits) / **cost** (rough).

### R1. Ship `StageIntrospect` interface + `explore-stage.mjs` driver

**What:** Optional interface `availableVerbs(state) / describe(state) / invokeVerb(id, state)`. CLI driver that loops through it. world-primary as first implementer. ActionSurface verbs become a *rendering* of `availableVerbs`.

**Why:** Collapses two problems into one: orphan verb-buttons (lens 3.8) and the mock-navigation tooling question. Makes the stage interaction graph queryable, which is the framework's core prescription.

**Who:** Tomás (verbs actually do things), Phase-5 verifier (can drive headless), Rin (a documented contract for new composers).

**Cost:** ~2 days. Interface + 1 example wiring + CLI + docs. world-primary verb fix is essentially free once the interface exists.

### R2. Reframe example renders as player-facing surfaces

**What:** For each of the 8 examples, replace JSON dumps with human-readable summaries. Hide tag strings behind display names. Color-code violations. Add a `(dev panel)` toggle that shows the current JSON view for stage-author debugging.

**Why:** Lens 3 finding — every render is a dev surface in a player's path. This is the highest-impact UX change; it's the difference between "looks like a debugger" and "looks like a game."

**Who:** Tomás (the entire reason he closes the tab in 60 seconds). Yuna (sees what good player UX looks like).

**Cost:** ~1 day per example, ~1 week total. Mechanical work; primitives unchanged.

### R3. README rewrite — front-door surface

**What:** Replace `README.md` with: 1-paragraph what-this-is, 1 screenshot/GIF of `yarn dev` showing the picker, 5-line quickstart, link to CLAUDE.md for the workflow detail. Add a "you'll see X" section for each phase. Mention `promote-example.mjs` as the exit ramp.

**Why:** Lens 1 finding — README is too thin to function as front door. README is Yuna's first 2 minutes; CLAUDE.md is her next 20.

**Who:** Yuna primarily; Rin if they discover the repo cold.

**Cost:** ~half a day (with screenshot capture).
