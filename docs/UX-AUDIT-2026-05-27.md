# UX Audit ‚Äî chub-stage-factory ‚Äî 2026-05-27

A three-lens UX audit of the chub-stage-factory repository, scoped to:

1. **Stage-author UX** ‚Äî someone (human or LLM) who clones this repo to ship one Chub stage. Walks through co-design, sees scripts, sees errors.
2. **Composer-author UX** ‚Äî someone extending `src/lib/patterns/` (or `src/lib/patterns/synergy/`). Has to learn the convention from existing files and produce one that fits.
3. **End-user UX** ‚Äî the player loading a deployed stage and taking their first turn. Studied through the 8 example stages.

Frameworks applied (see `~/git/rhizone/github-io/docs/{interaction-graph,affordance-opacity,affordance-surfaces,affordance-types}.md`):

- **Interaction graph** ‚Äî every affordance is an edge; failures are missing/hidden/orphaned edges, bottlenecks, sparse regions.
- **Affordance opacity** ‚Äî what the system hides about what you can do. Tribal knowledge cost.
- **Affordance surfaces** ‚Äî Miller's Law (‚â§7 simultaneous items) at every level. Removal over prioritization.
- **Affordance types** ‚Äî commands vs gestural vs ambient vs navigational vs directional vs data-entry. Different surfaces render different types.

The audit was specifically commissioned to answer: **what blocks Phase 5 manual browser verification?** That answer is in the Phase 5 Blocker List section. The rest of the audit is the substrate that makes that list legible.

## Personas

These three are derived from observable repository structure: the `CLAUDE.md` two-phase workflow (Phase 1 co-design / Phase 2 autonomous build), the `src/lib/patterns/` + `src/lib/patterns/synergy/` shape, and the 8 deployed `examples/`. Names are arbitrary; goals and contexts are not.

### P1. "Yuna, the weekend stage author"

Yuna found Chub a month ago, plays roleplay stages a few times a week, and watched a tutorial video on building one. She has just enough programming background to know what TypeScript is and how to run a yarn command. Saturday afternoon she clones chub-stage-factory because she wants to build "an inn-keeper stage where the player can buy drinks and ask gossip ‚Äî like the bar-keeper stage but in her own setting."

**Context:**
- Reads `README.md` first, then `CLAUDE.md` if she gets stuck.
- Will spend ~2 hours before deciding whether to continue or bounce.
- Wants to *see something on screen* in the first 20 minutes.
- Does not know what a "primitive" or a "composer" is; doesn't speak the library vocabulary.
- Will Cmd-click into example stage code to learn by example.
- Doesn't have a CHUB_AUTH_TOKEN, doesn't want to think about deployment yet.

**Goal:** A running local preview of a recognisable inn scene with one NPC and one verb that does something visible. From there, she'll iterate.

**What she'll touch:** `README.md`, `CLAUDE.md`, `DESIGN.md` (via Claude Code), `src/Stage.tsx`, `yarn dev` URL, eventually `examples/inventory/Stage.tsx` or `examples/world-primary/Stage.tsx` for reference.

### P2. "Rin, the composer hunter"

Rin is an experienced TypeScript developer who has built three Chub stages before, all by copy-pasting from `extension-template`. They found chub-stage-factory through a Discord link and were drawn in by the COMPOSITION.md framing ‚Äî "patterns instead of frameworks, primitives are orthogonal." They want to add a new composer: `patterns/conversation-tree.ts`, a dialogue-tree composer that wires `dialogue.ts` + `triggers.ts` into a Twine-shaped state machine. It's not in PATTERNS.md yet ‚Äî they think it should be, and want to test the claim that "if a thing reduces, it's permanently gone."

**Context:**
- Reads `src/lib/COMPOSITION.md`, `src/lib/README.md`, `src/lib/CONVENTIONS.md` before touching any code.
- Picks 3 existing composers to study the convention. Will their reading converge on a learnable pattern?
- Wants to PR upstream eventually but will keep a local branch first.
- Cares about `bun run lint`, `bun run test`, `bun run build` passing ‚Äî has seen messy repos before and is suspicious.
- Knows what an `@experimental` marker is and will check whether 0-caller primitives suggest a graveyard.

**Goal:** Ship a composer that (a) builds, (b) passes lint+test, (c) follows whatever the convention turns out to be, and (d) is sufficiently distinct that no existing composer subsumes it.

**What they'll touch:** `src/lib/patterns/synergy/cache-by-key.ts`, `src/lib/patterns/inventory.ts`, `src/lib/patterns/scene.ts` (for reference), then a new `src/lib/patterns/conversation-tree.ts` + an `examples/conversation-tree/` directory + an entry in `examples/registry.ts`.

### P3. "Tom√°s, the curious player"

Tom√°s opened the Chub homepage, browsed for stages, and the recommendation engine surfaced `world-primary` and `composite-showcase` (because they share tags with stages he played last week). He clicks `world-primary` first. He has never seen a stage that has its own UI (most stages he's played are pure prose-injection). He's going to give it 60 seconds to make sense or he'll click away.

**Context:**
- Brings zero technical background. Reads no docs.
- Sees the stage embedded in Chub's chat window. First impression is purely visual.
- Has muscle memory from text-only chat: type, press Enter, read reply.
- A button is a button. He'll click anything that looks clickable.
- Won't notice subtle UI affordances (resize handles, hover-only tooltips) unless they're labelled.
- If anything visibly fails ‚Äî empty panel, frozen button, garbage prose ‚Äî he bounces.

**Goal:** Have a couple of minutes of light fun. Maybe come back tomorrow if it sticks.

**What he'll touch:** the rendered DOM of `examples/world-primary/Stage.tsx` (or whichever stage Chub serves). Specifically the `ActionSurface` verbs, the `FreeformInput`, the `ScenePane`, the Chub-native message box.

### Findings ‚Äî Lens 1

**GREEN ‚Äî Phase 1 framing is correct.**

`CLAUDE.md` lines 67-85 frame Phase 1 as a co-design conversation that fills DESIGN.md before any code is written. The "bar for concrete enough" is named explicitly. Yuna does not need to know what a primitive is; Claude can name the relevant primitive for her based on her described mechanic. This is the highest-leverage UX choice in the repo and it works.

**GREEN ‚Äî Reference examples are properly indexed.**

`examples/README.md:45-56` lists all 8 examples with their mechanic and the PATTERNS.md recipe they exemplify. `examples/registry.ts:46-136` provides one-line descriptions of what each demonstrates. Yuna can scan this table in 30 seconds and pick the closest match. This is a *good affordance surface* per the surfaces framework ‚Äî 8 items is right at Miller's limit, the labels are human-readable, and the surface is one rendering of the underlying registry.

**YELLOW ‚Äî README is too thin to function as front door.**

`README.md` is 46 lines (`README.md:1-46`). It mentions "clone the repo, talk through Phase 1, run /loop" ‚Äî but for Yuna, this assumes she already knows what a slash command in Claude Code is. There is no minimal "what you'll see" ‚Äî no screenshot, no example output, no "if this works, you'll have X on screen." The README is a navigational affordance pointing into CLAUDE.md, but the destination is 266 lines and primarily addresses agents, not Yuna.

**YELLOW ‚Äî `DESIGN.md` skeleton has no examples in it.**

`DESIGN.md:1-83` is a blank-table skeleton. Yuna staring at "Tags" with no example values doesn't know what shape they should be (Chub tag slugs? freeform? both?). "Position: ADJACENT | NONE | COVER | FULLSCREEN" is a comment-hint; she has no idea what each visually means. The framework lens: this is a high-opacity data-entry surface. Fix would be a parallel `DESIGN.example.md` showing the *world-primary* DESIGN as a worked example.

**YELLOW ‚Äî `yarn dev` vs `bun run dev` vs `yarn install` inconsistency.**

`README.md:15-16` and `CLAUDE.md:153-158` both tell Yuna to run `yarn install` then `yarn dev`. `package.json:17-18` declares `bun scripts/run-stage.mjs` for smoke tests and `bun install` is referenced in DECISIONS.md ¬ß9. `flake.nix` (not read but referenced) provides `node@21.7.1 + yarn`. Yuna who follows the README will install yarn; she will not have bun, and the smoke runner will fail. The repo uses both. CI uses bun (DECISIONS.md ¬ß9). Local dev docs say yarn. Pick one.

**RED ‚Äî Build script error mode is destructive on interrupt.**

`scripts/build-example.mjs:36-66` performs:
1. `cpSync(publicDir, backup)` ‚Äî backup
2. `rmSync(publicDir, ...)` ‚Äî wipe public/
3. copy example's assets in
4. `vite build`
5. `finally { rmSync(public) + cpSync(backup, public) }` ‚Äî restore

If Yuna Ctrl-Cs during step 3 or 4, the `finally` runs and restores. Good. But if the **node process is killed (-9, OOM, crash)**, public/ is left wiped ‚Äî the next invocation will back up an empty directory, then permanently destroy the real public/ on the third invocation when temp dirs roll over. There is no checksum, no lock, no "is public/ a known template state?" check. For Yuna, this manifests as: ran build twice on a flaky laptop, public/ is now empty, fresh-clone needed. There's no STATUS.md trace of this hazard. Cite: `scripts/build-example.mjs:32-67`.

**RED ‚Äî Phase 2 has no human-facing progress signal.**

`CLAUDE.md:73-86` describes Phase 2 (`/loop`) as autonomous: it picks the next unchecked STATUS.md item, edits, builds, commits, pushes, repeats. Yuna is supposed to leave this running. But the *user-visible signal of progress* is: STATUS.md updates and git log. If she opens `yarn dev` mid-loop, she'll see whatever last-committed render exists ‚Äî possibly mid-feature, possibly broken-but-typechecks. There is no "Phase 2 is at task 4 of 9, current build status green" surface. Yuna will tab-out and tab back wondering if anything is happening. This is the *invisible state* failure mode from the interaction graph framework.

**RED ‚Äî `/loop` exit conditions are buried.**

`CLAUDE.md:85` says "Exit the loop when STATUS.md shows all tasks complete and the latest push passed the deploy workflow." Yuna does not know how to check the deploy workflow status. There is no `bun run status` or `node scripts/check-deploy.mjs` that tells her "you're done, or you're stuck on X." She has to know `gh run list` exists. This is tribal knowledge for power users.

**YELLOW ‚Äî Promote-example exists but isn't discoverable from the front door.**

`scripts/PROMOTE.md` describes the `promote-example.mjs` tool, which is genuinely Yuna's exit ramp: "I built my stage in this monorepo, now copy it out so I have a clean per-stage repo." But `README.md` doesn't mention it, `CLAUDE.md` doesn't mention it, `examples/README.md` doesn't mention it. The only way Yuna finds it is `ls scripts/` or grepping. Orphaned affordance.

**YELLOW ‚Äî `DESIGN.md` blank skeleton lives at the root.**

`DESIGN.md:1-83` is in the repo's root, alongside `STATUS.md`, `TODO.md`, `DECISIONS.md`, `CLAUDE.md`, `README.md`. Six top-level meta-files. Per Miller, this is at the upper bound; per the surfaces framework, it's a scanning-vs-searching boundary. A first-time visitor sees `ls` output and has to *read each filename* to triage. A subfolder `meta/` or naming convention (`PHASE1-DESIGN.md`, `PHASE2-STATUS.md`) would chunk visibly.

### Findings ‚Äî Lens 2

**GREEN ‚Äî Composer header comments are a strong convention.**

Every composer file under `src/lib/patterns/` opens with a doc-comment block following the schema `WHAT / WHY / SHAPE`. See `src/lib/patterns/inventory.ts:1-28`, `scene.ts:1-43`, `freeform-pipeline.ts:1-40`, `grafting.ts:1-21`. This is the convention Rin will pattern-match to. After reading 3, the shape is internalised. This is the **clean graph** outcome the interaction-graph framework praises: no filtering needed, the convention is the convention. The 33-composer batch (DECISIONS ¬ß13) maintained this; Rin can read any one and produce a 4th that fits.

**GREEN ‚Äî Synergy patterns have an enforced shared return shape.**

`src/lib/patterns/synergy/types.ts` declares `ComposedSubsystem<S>` with `state`, optional `contributors`, `hooks`, `shards`. All 22 synergy composers import this. `cache-by-key.ts:38-60` and `seed-from-player.ts:37-66` both return the same shape with a domain-specific extra method. Rin writing a 23rd synergy composer has a typed contract; if they don't return `ComposedSubsystem<‚Ä¶>`, TSC fails. This is *exactly* the affordance-types insight: when the type system enforces the surface, discovery becomes free.

**GREEN ‚Äî `bun run build` / `lint` / `test` / `test:smoke` exist and are wired into CI.**

`package.json:13-26` exposes all four. `.github/workflows/build-examples.yml` (not read this audit but per DECISIONS ¬ß9) runs all four. Rin's "did I break anything?" loop is fast and credible.

**YELLOW ‚Äî Top-level `src/lib/patterns/` (28 composers) violates Miller's Law as a surface.**

A flat directory listing of 28 composers ‚Äî `bulk-tick`, `dialogue`, `effects`, `faction`, `form`, `form-collection`, `grafting`, `inventory`, `lineage`, ‚Ä¶ ‚Äî exceeds the working-memory cap by 4√ó. Rin will search rather than scan. PATTERNS.md groups them by recipe number, which helps; but the file tree does not. Compare to the synergy subdir (22 files, also over) ‚Äî at least synergy has the type-shared `types.ts` to anchor on, plus the source-attribution comment ("Source: SillyTavern WI / NovelAI Lorebook keys / AID Story Cards" ‚Äî see `cache-by-key.ts:9-12`). The top-level patterns dir has no such grouping. A subfolder layout (`patterns/world/`, `patterns/combat/`, `patterns/character/`, `patterns/synergy/`) would chunk to ~7 per group.

**YELLOW ‚Äî Composer naming inconsistent between "extracted" and "newly built."**

DECISIONS ¬ß13 distinguishes extracted (lifted from examples) vs newly-built. Extracted ones (`inventory.ts`, `effects.ts`, `turn-combat.ts`, `body-transformation.ts`) return `*Bundle` types and use `pattern.buildBeforePrompt(msg, bound)`. Newly built ones like `grafting.ts` return a different-named object with `subsume`/`inject`/`replace` methods directly. Synergy ones return `ComposedSubsystem<S>`. This is **three return shapes for three vintages**. Rin asking "what shape should my composer return?" gets three valid answers. PATTERNS.md does not name the discriminant. See: `src/lib/patterns/inventory.ts:61-75` vs `src/lib/patterns/grafting.ts` vs `src/lib/patterns/synergy/types.ts:48-59`.

**YELLOW ‚Äî `withPersistence` HOC is the cleanest path but isn't documented at the top of patterns/.**

`examples/inventory/Stage.tsx:35` uses `extends withPersistence<‚Ä¶>()` ‚Äî a class-factory HOC. DECISIONS ¬ß1 says all 8 examples use it. But the docstring on `withPersistence` itself is in `src/lib/persistence/with-persistence.ts` (not read), and `src/lib/PATTERNS.md` ¬ß0 still shows the *raw* `PersistenceStore` + `bindStore` ceremony (`PATTERNS.md:22-65`), not the HOC. Rin reading PATTERNS.md will write 40 lines of persistence boilerplate that all 8 examples have already collapsed into one HOC call. The recipe doc is out of date relative to the examples.

**YELLOW ‚Äî `@experimental` markers are honest but mark dead-end edges.**

DECISIONS ¬ß7 marks `chat-window.ts`, `embeddings.ts`, `3d/scene.tsx`, `3d/loader.tsx`, `3d/use-three-handle.ts` as `@experimental ‚Äî used by 0-1 callers`. Rin building a new synergy composer that depends on `embeddings.ts` is building on a marked-experimental primitive whose own corrected-caller-count (DECISIONS ¬ß7 re-audit) is *zero*. There's no "do not depend on experimental" rule. The affordance-types lens: these are gestural affordances (low discoverability) mixed in with command affordances (high discoverability) on the same surface without distinction. Either commit (un-experimental) or delete; sitting in `@experimental` indefinitely is the worst of both worlds.

**RED ‚Äî PATTERNS.md does not list every shipped composer.**

`src/lib/patterns/` contains 28 composers; `src/lib/patterns/synergy/` contains 22. `PATTERNS.md` (read only the first 200 lines this audit) starts with recipes 0, 0a, 0b, 0c, 1, 2 ‚Äî recipe 1 is "Inventory" and points at raw `Inventory` primitive code, not `inventoryPattern`. Recipes 3-7 likely follow; whether each shipped composer has a documented recipe is unverified but doubtful given the 33-composer batch from a single afternoon. Rin trying to "find the recipe before writing the composer" hits a sparse documentation graph. Missing edges from PATTERNS.md to the composer files. Cite: `src/lib/PATTERNS.md:106-141` (recipe 1 shows raw primitive, doesn't mention `inventoryPattern`).

**RED ‚Äî Composer-author has no scaffolding tool.**

To add `conversation-tree.ts` Rin must by hand:
1. Create the file with the `WHAT/WHY/SHAPE` header.
2. Decide which return shape (`*Bundle` vs `ComposedSubsystem<S>` vs ad-hoc).
3. Wire imports from `../intent`, `../trigger`, `../timeline`, etc.
4. Create `examples/conversation-tree/Stage.tsx` + `test-init.json` + `chub_meta.yaml` + `scenario.yaml` + `characters/` (per `examples/README.md:24-28`).
5. Add entry to `examples/registry.ts:46-136`.
6. Update `PATTERNS.md` (if maintained), `REFERENCE.md`, `ROADMAP.md`.

No `bun run new:pattern <name>` or `node scripts/scaffold-composer.mjs`. This is reasonable for a single-author repo; for a "library positioning itself as accepting community composers," it's an undocumented barrier. See `scaffolding/` directory in the ecosystem-docs `CLAUDE.md` for the pattern that *should* exist here.
